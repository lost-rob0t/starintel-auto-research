* Purpose

This project index[fn:project-index] tracks Star-Lang[fn:star-lang] language research without treating an unfinished prototype as an approved implementation design.

* Canonical Source

- [[file:../../design/STAR-00-Starintel-4th-generation-Ideas.org][STAR-00 StarIntel Fourth Generation Ideas]]

The source backlog requires complete Common Lisp[fn:common-lisp], Racket[fn:racket], and Scheme[fn:scheme] prototypes of the same workflow before any language is recommended.

* Completed Foundation

- [[file:../../research/star-lang/STAR-LANG-RESEARCH-000-language-design-foundations.org][STAR-LANG-RESEARCH-000 Star-Lang Language Design Foundations]]

Status: completed research. It defines the bounded language boundary, closed core, effects, capabilities, durable run semantics, actor envelopes, Prolog routing, conditions, restarts, replay, testing, and the later design handoff. It is not an approved implementation design.

* Active Comparative Loop

- [[file:../../research/star-lang/STAR-LANG-RESEARCH-001-common-lisp-comparative-baseline.org][STAR-LANG-RESEARCH-001 Common Lisp Comparative Baseline]]

Status: review. The loop establishes the shared fixture, Common Lisp declaration macros, normalized dataflow plan, deterministic fake actor, declared fake-agent tool, persistence checks, tests, and benchmark harness.

* Active Specification and Domain Loop

- [[file:../../research/star-lang/STAR-LANG-RESEARCH-006-specification-libraries-and-domain-servers.org][STAR-LANG-RESEARCH-006 Specification Libraries and Domain Servers]]

Status: review. This requirements loop defines digest-locked data-only specification imports, additive schema extension, typed source/predicate/destination relations, immutable dataset views, typed actors, keyed domain servers, graph indexes, and the resolver/compiler/runtime boundary. It does not bypass the comparative evaluation or promotion gate.

* Active Compiler and Runtime-Binding Loop

- [[file:../../research/star-lang/STAR-LANG-RESEARCH-007-normalized-ir-and-cl-gserver-binding.org][STAR-LANG-RESEARCH-007 Normalized IR and cl-gserver Binding]]

Status: review. This pass separates comparative Common Lisp surface declarations from a closed normalized IR and a declarative cl-gserver adapter. Core IR retains runtime-neutral =:send= nodes; only the optional cl-gserver binding lowers them to =:tell= and records =:actor-of= construction. It preserves the immutable local specification graph and keyed domain-server requirements from Loop 006.

* Active Core Surface and Wire Loop

- [[file:../../research/star-lang/STAR-LANG-RESEARCH-008-core-surface-and-wire-protocol.org][STAR-LANG-RESEARCH-008 Core Surface and Wire Protocol]]

Status: review. This pass makes =spec-library=, =scalar=, =enum=, =document=, =predicate=, and =message= the concrete source surface, qualifies local types into normalized schema IR, validates inheritance and actor contracts, and defines matching native and external actor contracts. A portable =star-message-v1= manifest exposes the same contracts to non-Common-Lisp actors.

* Active Canonical JSON and Binding Loop

- [[file:../../research/star-lang/STAR-LANG-RESEARCH-009-canonical-json-and-generated-bindings.org][STAR-LANG-RESEARCH-009 Canonical JSON and Generated Bindings]]

Status: review. This pass serializes the validated portable manifest and message envelopes as deterministic JSON and generates Python and TypeScript bindings from the same normalized contracts. Decimal values cross the wire as strings, exact Star-Lang field names are preserved, and no adapter parses the original specification independently.

* Active Message Lifecycle Loop

- [[file:../../research/star-lang/STAR-LANG-RESEARCH-010-message-lifecycle-and-delivery-semantics.org][STAR-LANG-RESEARCH-010 Message Lifecycle and Delivery Semantics]]

Status: review. This pass defines commands, events, replies, acknowledgements, errors, and cooperative cancellation with explicit correlation, causation, attempts, deadlines, and idempotency. It assumes at-least-once delivery where transports redeliver and deliberately does not claim exactly-once execution.

* Active Deterministic Dispatcher Loop

- [[file:../../research/star-lang/STAR-LANG-RESEARCH-011-deterministic-dispatcher.org][STAR-LANG-RESEARCH-011 Deterministic Dispatcher]]

Status: review. This pass executes the lifecycle model in a pure FIFO dispatcher with manifest-checked actor routing, terminal outcome replay, in-progress duplicate suppression, retry redelivery, deterministic deadlines, and cancellation. It is also the standalone execution baseline.

* Active Transport Settlement Loop

- [[file:../../research/star-lang/STAR-LANG-RESEARCH-012-transport-port-and-settlement.org][STAR-LANG-RESEARCH-012 Transport Port and Settlement]]

Status: review. This pass separates actor-visible lifecycle acknowledgements from broker settlement, defines a transport-neutral port, models delayed at-least-once redelivery and dead-letter rejection, and adds a resumable publication outbox so outputs complete before the source delivery is settled.

* Active Optional Runtime Adapter Loop

- [[file:../../research/star-lang/STAR-LANG-RESEARCH-013-cl-gserver-runtime-facade.org][STAR-LANG-RESEARCH-013 Optional cl-gserver Runtime Adapter]]

Status: review. This pass proves native Star-Lang actors still execute directly with ordinary handler functions and no actor system. It separately binds selected native actors to asynchronous cl-gserver =actor-of= and =tell= operations and defines an optional runtime-directory port for actor existence and liveness data.

The email-enumeration fixture is the conformance fixture[fn:conformance-fixture] for every language implementation. Later prototypes may decode it differently but may not change its workflow, fake actor outcomes, persistence expectations, or final result.

The specification/domain prototype is a parallel requirements fixture. Equivalent semantics must be carried into the language candidates after each baseline passes; it does not replace the original conformance fixture.

The normalized-IR prototype is a compiler-boundary fixture. Equivalent frontends must be able to produce the same runtime-neutral graph, while runtime bindings remain replaceable projections.

The FEC core fixture is the concrete schema, wire, lifecycle, dispatcher, standalone-execution, transport, and optional-runtime fixture. Equivalent frontends must preserve the same contracts and lifecycle outcomes regardless of whether handlers execute directly, through cl-gserver, or through an external actor adapter.

* Ordered Loops

** DONE Loop 000: Language Design Foundations

Define the minimum safe source, checked intermediate representation[fn:ir], effects, durable execution, recovery, actor, reasoning, and testing architecture.

** REVIEW Loop 001: Common Lisp Baseline

Implement the required email-enumeration workflow with ordinary macros and no reader macros or unrestricted =EVAL=.

** TODO Loop 002: Racket Embedded Baseline

Implement the same fixture as an embedded Racket domain-specific language[fn:dsl].

** TODO Loop 003: Racket =#lang starintel= Baseline

Implement the same fixture through a dedicated Racket language boundary.

** TODO Loop 004: Scheme Baseline

Implement the same fixture in one practical Scheme implementation while recording which parts remain portable.

** TODO Loop 005: Comparative Evaluation

Run equivalent tests and controlled benchmarks, score each required category, record unresolved risks, and produce a recommendation only after every prototype is complete.

** REVIEW Loop 006: Specification Libraries and Domain Servers

Define immutable import resolution, lock graphs, additive document extension, canonical relation traversal, typed actor contracts, and keyed domain-server manifests.

** REVIEW Loop 007: Normalized IR and cl-gserver Binding

Separate normalized compiler output from runtime-specific binding. Keep =send= in core IR and lower it to =tell= only in the optional cl-gserver adapter.

** REVIEW Loop 008: Core Surface and Wire Protocol

Compile concrete specification syntax into qualified schema IR, validate schema and actor semantics, and emit portable native and external actor contracts.

** REVIEW Loop 009: Canonical JSON and Generated Bindings

Serialize one validated manifest into deterministic JSON and generate Python and TypeScript bindings while preserving exact field names and precision.

** REVIEW Loop 010: Message Lifecycle and Delivery Semantics

Define correlation, causation, accepted versus completed acknowledgements, retryable errors, cooperative cancellation, attempts, deadlines, and idempotency without claiming exactly-once execution.

** REVIEW Loop 011: Deterministic Dispatcher

Execute lifecycle rules in a pure state machine and provide the standalone native-handler execution baseline.

** REVIEW Loop 012: Transport Port and Settlement

Separate lifecycle outcomes from broker settlement and resume interrupted output publication without repeating actor work.

** REVIEW Loop 013: Optional cl-gserver Runtime Adapter

Preserve standalone execution, optionally delegate selected native handlers to asynchronous cl-gserver actors, and expose actor topology and liveness through a separate optional runtime directory.

* Comparison Invariants

Every prototype must preserve:

1. the same document schemas
2. the same =Target= and nested =User= input
3. the same email domains
4. the same fake actor results
5. the same agent and declared tool behavior
6. the same named dataflow stages
7. the same transient and persistent document policy
8. the same final review
9. inspectable expansion or compiled forms
10. runnable tests and benchmark entry points

After a language baseline passes, its next requirements fixture must also preserve:

1. exact version and digest import matching
2. additive-only schema extension
3. explicit source, predicate, and destination relation positions
4. field access without dot syntax
5. destination-organization filtering
6. typed actor input and output contracts
7. keyed domain-server ownership, indexes, capabilities, and restart policy
8. a runtime-neutral normalized graph before adapter binding
9. =send= in core IR and runtime-specific message operations only in adapters
10. deterministic node identities suitable for execution, diagnostics, and visualization
11. the concrete =spec-library= schema surface
12. qualified portable type and message identifiers
13. compatible native and external actor contracts
14. a versioned language-neutral message envelope
15. semantic validation before adapter generation
16. byte-stable canonical JSON
17. exact field names across generated languages
18. precision-safe decimal wire values
19. correlation and direct causation identifiers
20. command idempotency scoped by actor and message type
21. accepted and completed acknowledgement distinction
22. explicit retryable versus terminal errors
23. cooperative cancellation semantics
24. no exactly-once delivery claim
25. deterministic FIFO command execution
26. terminal duplicate replay without handler reinvocation
27. in-progress duplicate suppression
28. retry redelivery with incremented attempt and stable correlation
29. deadline failure before actor execution
30. cancellation indexed by message and correlation
31. actor-visible lifecycle acknowledgements separated from transport settlement
32. all required lifecycle outcomes published before source delivery settlement
33. resumable ordered publication records for interrupted output delivery
34. held commands remain unsettled until cancellation outcomes publish
35. poison inputs terminate through reject and dead-letter behavior
36. broker redelivery count remains distinct from Star-Lang attempt count
37. native Star-Lang execution remains valid without cl-gserver
38. runtime adapters remain replaceable and optional
39. direct and adapted native execution preserve the same lifecycle contracts
40. runtime actor discovery and liveness remain separate from language execution
41. cl-gserver actor existence may inform routing but cannot define program validity
42. late adapted actor results cannot reopen terminal commands

* Promotion Gate

No prototype enters =roam/implement/= and no Star-Lang implementation design is approved from this index. The later design gate[fn:design-gate] may open only after Loop 005 records the complete evidence and nsaspy approves a numbered design. Loops 006 through 013 add requirements but do not independently authorize promotion.

* Footnotes and Glossary

[fn:project-index] Project index: A maintained navigation file that lists the authoritative notes, current work, dependencies, ordering, and decision gates for one project area.

[fn:star-lang] Star-Lang: The proposed StarIntel language for declaring typed document flows, actor work, agent tools, persistence, recovery, importable specifications, domain servers, resumable analysis behavior, portable actor contracts, lifecycle semantics, deterministic standalone execution, optional runtime adapters, and transport settlement.

[fn:common-lisp] Common Lisp: A standardized Lisp programming language with functions, macros, object facilities, hash tables, and a structured condition and restart system.

[fn:racket] Racket: A Lisp-family language and language-building platform that supports embedded languages and dedicated =#lang= module languages.

[fn:scheme] Scheme: A Lisp-family language standardized around a small core, lexical scope, procedures, and macros, with multiple practical implementations.

[fn:conformance-fixture] Conformance fixture: One reviewed input and expected-output dataset used unchanged across implementations so their behavior can be compared directly.

[fn:ir] Intermediate representation, or IR: A compiler-owned data structure between source syntax and runtime execution.

[fn:dsl] Domain-specific language, or DSL: A deliberately limited language designed for one problem area instead of general-purpose programming.

[fn:design-gate] Design gate: The explicit review point that decides whether research is complete enough to freeze interfaces in a numbered design.
