* Purpose

This project index[fn:project-index] tracks Star-Lang[fn:star-lang] language research without treating an unfinished prototype as an approved implementation design.

The email-enumeration fixture is the conformance fixture[fn:conformance-fixture] for every language implementation. Later prototypes may decode it differently but may not change its workflow, fake actor outcomes, persistence expectations, or final result.

The specification/domain prototype is a parallel requirements fixture. Equivalent semantics must be carried into the language candidates after each baseline passes; it does not replace the original conformance fixture.

The normalized-IR prototype is a compiler-boundary fixture. Equivalent frontends must be able to produce the same runtime-neutral graph, while runtime bindings remain replaceable projections.

* Ordered Loops

** DONE Loop 000: Language Design Foundations

Define the minimum safe source, checked intermediate representation[fn:ir], effects, durable execution, recovery, actor, reasoning, and testing architecture.

** REVIEW Loop 001: Common Lisp Baseline

Implement the required email-enumeration workflow with ordinary macros and no reader macros or unrestricted =EVAL=.

** TODO Loop 002: Racket Embedded Baseline

Implement the same fixture as an embedded Racket domain-specific language[fn:dsl].

** TODO Loop 003: Racket =#lang starintel= Baseline

Implement the same fixture through a dedicated Racket language boundary.

** TODO Loop 004: Scheme Baseline

Implement the same fixture in one practical Scheme implementation while recording which parts remain portable.

** TODO Loop 005: Comparative Evaluation

Run equivalent tests and controlled benchmarks, score each required category, record unresolved risks, and produce a recommendation only after every prototype is complete.

** REVIEW Loop 006: Specification Libraries and Domain Servers

Define immutable import resolution, lock graphs, additive document extension, canonical relation traversal, typed actor contracts, and keyed domain-server manifests. This loop elaborates requirements in parallel and does not change the order or decision authority of Loops 002 through 005.

** REVIEW Loop 007: Normalized IR and cl-gserver Binding

Separate the Common Lisp surface experiment from normalized compiler output and runtime-specific actor binding. Keep =send= in core IR, bind it to cl-gserver =tell= only after compilation, and represent domain servers as keyed aggregate authorities. This parallel loop does not change the comparison order or authorize implementation promotion.

* Comparison Invariants

Every prototype must preserve:

1. the same document schemas
2. the same =Target= and nested =User= input
3. the same email domains
4. the same fake actor results
5. the same agent and declared tool behavior
6. the same named dataflow stages
7. the same transient and persistent document policy
8. the same final review
9. inspectable expansion or compiled forms
10. runnable tests and benchmark entry points

After a language baseline passes, its next requirements fixture must also preserve:

1. exact version and digest import matching
2. additive-only schema extension
3. explicit source, predicate, and destination relation positions
4. field access without dot syntax
5. destination-organization filtering
6. typed actor input and output contracts
7. keyed domain-server ownership, indexes, capabilities, and restart policy
8. a runtime-neutral normalized graph before adapter binding
9. =send= in core IR and runtime-specific message operations only in adapters
10. deterministic node identities suitable for execution, diagnostics, and visualization

* Promotion Gate

No prototype enters =roam/implement/= and no Star-Lang implementation design is approved from this index. The later design gate[fn:design-gate] may open only after Loop 005 records the complete evidence and nsaspy approves a numbered design. Loops 006 and 007 add requirements but do not independently authorize promotion.

* Footnotes and Glossary

[fn:project-index] Project index: A maintained navigation file that lists the authoritative notes, current work, dependencies, ordering, and decision gates for one project area.

[fn:star-lang] Star-Lang: The proposed StarIntel language for declaring typed document flows, actor work, agent tools, persistence, recovery, importable specifications, domain servers, and resumable analysis behavior.

[fn:common-lisp] Common Lisp: A standardized Lisp programming language with functions, macros, object facilities, hash tables, and a structured condition and restart system.

[fn:racket] Racket: A Lisp-family language and language-building platform that supports embedded languages and dedicated =#lang= module languages.

[fn:scheme] Scheme: A Lisp-family language standardized around a small core, lexical scope, procedures, and macros, with multiple practical implementations.

[fn:conformance-fixture] Conformance fixture: One reviewed input and expected-output dataset used unchanged across implementations so their behavior can be compared directly.

[fn:ir] Intermediate representation, or IR: A compiler-owned data structure between source syntax and runtime execution.

[fn:dsl] Domain-specific language, or DSL: A deliberately limited language designed for one problem area instead of general-purpose programming.

[fn:design-gate] Design gate: The explicit review point that decides whether research is complete enough to freeze interfaces in a numbered design.
