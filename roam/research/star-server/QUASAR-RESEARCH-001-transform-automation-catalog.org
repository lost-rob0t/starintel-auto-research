* Corrected Ownership Boundary

Quasar owns graph actions, transforms, action discovery, action invocation, graph-operation batches, renderer integration, and undo grouping.

Star-Lang does not provide Quasar transforms or context-menu actions. Star-Lang owns typed document and relation specifications, specification libraries, datasets, actors, declarative playbooks, and resumable analysis-run semantics. A Star-Lang playbook may invoke a registered Quasar action through an explicit adapter, but the action remains a Quasar extension or external integration.

Any older transform wording in this note must be read under this boundary. This catalog is an optional Quasar extension backlog, not a requirement for the basic graph editor MVP.

* Decision

Quasar should adopt Maltego's strongest interaction model—typed entities, transforms filtered by selected input types, transform sets, and machines—while making automation more complete and more programmable.

Transforms are Quasar-owned extensions or external integrations. They may run interactively, in bulk, on schedules, from graph conditions, from external events, or when invoked by a Star-Lang playbook through an adapter. Their results return through the Quasar action API and are applied to the graph with run metadata.

The user does not need to approve every transform result. Human control is provided through transform selection, configuration, scheduling, limits, cancellation, rollback, result editing, and optional explicit pause points.

* Maltego Patterns to Preserve and Extend

Maltego provides several durable ideas:

- entities determine which transforms are available;
- transforms map one or more typed inputs into typed graph results;
- transform sets organize large catalogs;
- machines compose transforms sequentially and in parallel;
- graph selection can be supplied as a transform input;
- custom entities and transforms extend the system;
- transform results can create multiple entities and links.

Quasar should extend those ideas with:

- first-class batch and whole-selection transforms;
- streaming outputs instead of one final response;
- optional durable run state and checkpoint replay after the editor MVP;
- optional triggers and feeders implemented only when a concrete integration requires them;
- optional automatic continuation policies with explicit bounds;
- typed cost, fan-out, cache, idempotency, and rollback contracts;
- interoperability with Star-Lang typed data and optional playbook orchestration, plus adapters to existing services where useful;
- deterministic result identity and deduplication across repeated runs;
- evidence and provenance attached to every generated claim and relationship.

* Transform Kinds

** Entity transform

Runs against one entity or canonical object.

Examples:

- domain to DNS records;
- email address to accounts;
- document to extracted entities;
- organization to known domains.

** Batch transform

Runs once against a homogeneous or heterogeneous selection rather than dispatching one isolated request per node.

Examples:

- resolve 500 domains in one DNS batch;
- compare selected people across datasets;
- hash and classify a document set;
- calculate shared infrastructure among selected organizations.

** Graph transform

Receives a bounded graph projection including selected nodes, edges, claims, and layout-independent identifiers.

Examples:

- shortest evidentiary path;
- community detection;
- common-neighbor expansion;
- temporal path comparison;
- identify bridge nodes across clusters.

** Stream transform

Emits results incrementally and remains observable while running.

Examples:

- web crawl;
- large document extraction;
- continuous feed ingestion;
- paginated social or registry collection;
- long-running enrichment across a list.

** Feeder

Creates initial entities or claims from an external or scheduled source.

Examples:

- RSS and Atom feed;
- saved-query results;
- watched directory;
- message queue;
- recurring API poll;
- imported CSV, JSON, JSONL, STIX, or FollowTheMoney data.

** Trigger

Starts an action or playbook when a graph, workspace, run, schedule, or message condition becomes true.

Examples:

- when a new domain is added, resolve DNS and certificates;
- when a new document arrives, extract entities and claims;
- when a monitored page changes, capture and diff it;
- when an actor emits a retryable error, route it to a fallback actor;
- when two clusters become connected, recompute paths and summaries.

** Machine / playbook

A Quasar machine or a Star-Lang playbook may compose registered Quasar transforms with branches, loops, concurrency, retries, checkpoints, cancellation, restart handling, and optional pause points. Star-Lang owns orchestration semantics, not transform definitions.

* Transform Contract

Every Quasar transform definition should declare:

- stable qualified identifier and semantic version;
- display name, description, tags, aliases, and transform sets;
- accepted input types and selection cardinality;
- produced object, claim, relationship, evidence, and event types;
- parameter schema and defaults;
- precondition predicate;
- implementation adapter: in-process function, subprocess, HTTP integration, existing actor or service, or optional domain server;
- required capabilities and credentials;
- network, filesystem, process, database, and model effects;
- direct-persistence behavior;
- merge, conflict, replacement, and deduplication behavior;
- idempotency scope and deterministic result-key function;
- cache policy and freshness window;
- pagination and continuation-token behavior;
- expected fan-out and hard result limit;
- concurrency, rate, time, memory, and monetary limits;
- retry and backoff policy;
- cancellation semantics;
- rollback or compensating action when meaningful;
- provenance and evidence production;
- streaming event types;
- whether the transform is safe for automatic continuation;
- optional pause points or intervention conditions.

A transform catalog must be queryable by input type, output type, tag, cost, capability, expected yield, implementation, freshness, and automation safety.

Transform discovery should rank actions using:

1. exact input-type match;
2. selection-cardinality match;
3. current workspace and dataset availability;
4. required capabilities and credentials;
5. expected novelty relative to existing results;
6. expected result yield and fan-out;
7. freshness of existing equivalent results;
8. runtime and monetary cost;
9. user history and pinned transforms;
10. configured automation policies.

The menu should show why a transform is unavailable and should support fuzzy search across names, aliases, tags, accepted types, and produced types.

* Automatic Expansion Engine

Automatic expansion should be a policy-driven actor, not a hard-coded recursive graph callback.

For every newly persisted object or claim, the expansion actor may evaluate eligible transforms and enqueue work when all policy predicates pass.

Required controls:

- maximum expansion depth;
- maximum new objects and claims per root;
- maximum fan-out per transform;
- maximum concurrent runs;
- maximum runtime and monetary cost;
- allowed and denied transform sets;
- allowed input and output types;
- dataset and workspace scope;
- freshness and cache requirements;
- stop conditions;
- deduplication key;
- cycle detection;
- novelty threshold;
- confidence threshold when a transform consumes inferred claims;
- per-domain or per-source rate limits;
- cancellation propagation;
- rollback grouping.

** Continuation modes

- =off=: never continue automatically;
- =one-hop=: execute configured transforms only on the initial selection;
- =bounded=: continue until depth, fan-out, novelty, cost, or time limits are reached;
- =playbook=: continue according to an explicit Star-Lang playbook;
- =triggered=: continue only when declared graph or event predicates match;
- =continuous=: keep a feeder, query, or alert active and execute downstream actions on new results.

** Novelty and duplicate suppression

Before enqueueing a transform, the actor should calculate a deterministic invocation key over:

- transform identifier and version;
- normalized input identifiers;
- relevant parameter values;
- dataset and policy scope;
- freshness bucket.

It should suppress:

- exact terminal duplicates within the idempotency window;
- runs whose outputs are still fresh;
- recursive cycles that do not add new typed information;
- fan-out branches whose expected novelty is below policy threshold.

* Proposed Machines / Star-Lang Playbooks

** Domain footprint

1. normalize the domain;
2. resolve DNS records;
3. discover subdomains;
4. resolve addresses, ASNs, and netblocks;
5. collect certificates and certificate-linked domains;
6. discover HTTP services;
7. capture response metadata, technologies, links, scripts, and screenshots;
8. extract script-referenced endpoints;
9. group shared infrastructure;
10. persist a bounded graph projection and run summary.

** Organization map

1. resolve organization identifiers;
2. collect domains, filings, addresses, officers, subsidiaries, and public accounts;
3. expand domains into infrastructure and web properties;
4. extract people and organizations from official documents;
5. cross-reference identifiers and addresses;
6. build ownership, role, infrastructure, and document projections.

** Person identity graph

1. normalize the name and known identifiers;
2. generate alias and account candidates;
3. collect public profile snapshots and referenced URLs;
4. find organization roles and authored or mentioning documents;
5. score identity-resolution features;
6. automatically merge only when configured thresholds pass;
7. retain ambiguous candidates as separate objects with match evidence.

** Social-account expansion

1. capture profile state;
2. collect bounded recent posts;
3. extract mentions, replies, quotes, hashtags, media, and external links;
4. resolve linked domains and accounts;
5. build interaction and timeline projections;
6. schedule incremental refresh from the last continuation cursor.

** Document-set analysis

1. hash and identify files;
2. extract metadata and text;
3. detect language and document structure;
4. extract typed entities and anchored claims;
5. create citation and co-occurrence graphs;
6. compare claims across documents;
7. generate summaries linked to source passages;
8. export an evidence package.

** Bug-bounty surface map

1. import declared scope;
2. expand wildcard domains and known assets;
3. resolve DNS, certificates, hosts, and HTTP services;
4. crawl bounded web surfaces;
5. extract scripts, endpoints, forms, and parameters;
6. identify technology and historical observations;
7. deduplicate shared infrastructure;
8. stream new assets and observations into the workspace;
9. schedule refresh according to asset freshness.

** Continuous change watch

1. run a saved query or feeder on schedule;
2. compare results against the prior checkpoint;
3. persist additions, removals, and changed claims;
4. capture changed source material;
5. execute configured downstream transforms;
6. emit an alert and update the workspace timeline.

#+begin_src lisp
(quasar-transform dns-records
  (:sets infrastructure dns automation)
  (:accepts (domain :cardinality one-or-many))
  (:produces dns-record dns-name ip-address claim evidence)
  (:actor dns-collector)
  (:persistence direct)
  (:idempotency
   (:key (transform-version normalized-inputs parameters freshness-bucket)))
  (:cache (:ttl "15m"))
  (:fanout (:expected 12 :maximum 4096))
  (:limits (:timeout "2m" :concurrency 32))
  (:retry (:maximum-attempts 4 :backoff exponential))
  (:automatic-continuation t))

(quasar-transform crawl-site
  (:sets web automation bug-bounty)
  (:accepts (url :cardinality one-or-many))
  (:produces url document form endpoint script relationship evidence)
  (:actor web-crawler)
  (:streaming t)
  (:persistence direct)
  (:checkpoint (:every-results 100 :every "30s"))
  (:fanout (:maximum 10000))
  (:limits (:timeout "30m" :concurrency 8))
  (:cancellation cooperative))

(playbook domain-footprint
  (:input root domain)
  (:persistence direct)
  (:interruptible t)
  (:limits (:maximum-depth 4
            :maximum-results 50000
            :maximum-runtime "2h"))
  (parallel
    (run dns-records root :as dns)
    (run registration-records root :as registration)
    (run domain-certificates root :as certificates))
  (checkpoint initial-infrastructure)
  (run discover-http-services dns.addresses :as web)
  (parallel-for-each service in web.services
    (:concurrency 16)
    (run inspect-web-service service))
  (run shared-infrastructure
       (outputs dns registration certificates web)))

(trigger expand-new-domain
  (:when (and (event-type-p event 'object-created)
              (object-type-p event.object 'domain)
              (workspace-policy-p event.workspace 'auto-expand-domains)))
  (:run domain-footprint event.object))
#+end_src

* Implementation Order

** Phase 0: protocol

- freeze transform manifest fields;
- map transform invocation onto the existing Star-Lang message lifecycle;
- define generated Common Lisp, Python, and TypeScript bindings;
- define deterministic invocation and result keys;
- define streaming event and checkpoint envelopes.

** Phase 1: useful vertical slice

Implement:

- domain to DNS records;
- DNS name to addresses;
- address to ASN and netblock;
- host to HTTP services;
- URL to response metadata;
- URL to links and scripts;
- document to entities;
- evidence capture;
- common-neighbor analysis;
- one =domain-footprint= playbook.

** Phase 2: automation

- automatic expansion actor;
- transform ranking and search;
- feeders and triggers;
- batch and graph transforms;
- scheduled and continuous playbooks;
- cancellation and rollback groups.

** Phase 3: broad catalog

- organization and person transforms;
- social platform adapters;
- geospatial and temporal transforms;
- cryptocurrency transforms;
- entity resolution;
- language-model transforms;
- extension-distributed transform packages.
