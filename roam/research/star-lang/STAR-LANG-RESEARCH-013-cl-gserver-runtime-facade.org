#+title: STAR-LANG-RESEARCH-013 Optional cl-gserver Runtime Adapter
#+description: Preserve standalone Star-Lang execution while optionally binding native actor contracts to asynchronous cl-gserver mailboxes and runtime liveness discovery.
#+status: REVIEW
#+filetags: :starintel:star-lang:cl-gserver:sento:actors:runtime:standalone:directory:liveness:dispatcher:transport:outbox:common-lisp:research:

* Purpose

Star-Lang is not a cl-gserver language and does not require an actor system to execute.

Loop 011 already proves standalone execution through the deterministic dispatcher and ordinary registered Common Lisp handler functions. Loop 012 adds an optional transport boundary. This pass adds cl-gserver as one replaceable execution adapter for native actors.

The invariant is:

#+begin_quote
Star-Lang owns language semantics. cl-gserver may execute actor work and report runtime topology, but it does not define whether a valid Star-Lang program can run.
#+end_quote

* Execution Modes

The same validated Star-Lang manifest supports multiple execution modes.

** Standalone in-process execution

The dispatcher directly invokes registered Common Lisp handler functions.

#+begin_src text
command
  -> deterministic dispatcher
  -> registered handler function
  -> lifecycle outcomes
#+end_src

This mode requires no cl-gserver actor system, coordinator actor, actor mailbox, actor registry, or runtime discovery service.

It is appropriate for:

- command-line execution
- tests
- small local ingest jobs
- embedded use
- batch processing
- environments where actor concurrency is unnecessary

** cl-gserver-backed execution

The dispatcher delegates native handler execution to cl-gserver actors through asynchronous =tell= operations.

#+begin_src text
command
  -> deterministic dispatcher proxy
  -> tell native actor
  -> native handler
  -> tell runtime coordinator
  -> finish deferred dispatcher command
  -> lifecycle outcomes
#+end_src

This mode adds actor isolation, mailbox execution, runtime identity, topology, and liveness information while preserving the same Star-Lang contracts and outcomes.

** External actor execution

Portable actor contracts may also be implemented by external processes through the Star message protocol. cl-gserver is not involved unless it is intentionally used as a local bridge or supervisor.

* Ownership Boundary

Star-Lang owns:

- specification libraries and typed documents
- actor input and output contracts
- command validation
- lifecycle envelopes
- correlation, causation, attempts, and idempotency
- accepted, completed, retry, failure, and cancellation semantics
- deterministic dispatcher state
- publication outbox ordering
- transport settlement policy

cl-gserver may own:

- actor construction
- actor paths and runtime references
- asynchronous mailbox delivery
- actor running state
- actor-context membership
- local actor lookup
- dispatcher threads and mailbox scheduling
- actor stop and actor-system shutdown

The optional adapter may use cl-gserver information for routing and operations, but a missing cl-gserver system is not a language error.

* cl-gserver Semantics Used

Sento documents =actor-of= as actor construction under an actor context with a =:receive= message-processing function. It documents =tell= as asynchronous fire-and-forget delivery whose receive return value is ignored.

Sento also documents:

- =all-actors= for retrieving actors in an actor context
- =find-actors= for path-based actor lookup
- =running-p= for checking whether an actor cell is running
- actor-system registration that tracks which actors exist

Source:

- [[https://mdbergmann.github.io/cl-gserver/index.html][Sento reference manual]]

The adapter therefore uses asynchronous jobs and a coordinator result message rather than treating =tell= as a function call.

* Execution Port

The optional cl-gserver execution port exposes:

1. =actor-of= to construct an actor under a context
2. =tell= to enqueue one message asynchronously
3. =stop= to stop an actor
4. =shutdown= to shut down the actor context or system

A later concrete Sento adapter may call:

#+begin_src lisp
(ac:actor-of system
  :name actor-name
  :receive receive-function)

(act:tell actor message sender)
#+end_src

The Star-Lang adapter itself does not require Sento packages during the comparative conformance pass.

* Optional Runtime Directory

Runtime discovery is separate from execution semantics.

The runtime-directory port returns normalized actor observations such as:

#+begin_src lisp
((:name "fec-native-importer"
  :runtime :cl-gserver
  :alive t
  :path "/user/fec-native-importer")
 (:name "stopped-worker"
  :runtime :cl-gserver
  :alive nil
  :path "/user/stopped-worker"))
#+end_src

A concrete cl-gserver directory adapter can build this snapshot using =ac:all-actors= or =ac:find-actors= and =act-cell:running-p=.

The directory can support:

- determining which declared actors currently exist
- selecting an available local actor
- displaying runtime health in Quasar
- supervisor and operations views
- avoiding sends to actors already known to be stopped
- comparing declared actor contracts with observed runtime topology

The runtime directory is optional. Standalone Star-Lang execution does not construct or query one.

* Native Actor Contracts

The optional adapter accepts compiled native actor declarations such as:

#+begin_src lisp
(actor fec-native-importer
  (:runtime native
   :accepts (ingest-page)
   :produces (index-fec-record)
   :handler fec-native-handler
   :restart permanent
   :mailbox (bounded 128)))
#+end_src

Before actor construction it verifies:

- the declaration is a compiled native actor contract
- the actor appears as native in the portable manifest
- the named handler has been registered as a Common Lisp function
- the runtime adapter and transport adapter share the same deterministic dispatcher

The same native contract may instead be registered directly with the standalone dispatcher. The contract does not imply cl-gserver execution.

* Actor Topology

When the optional adapter is started, it creates:

- one coordinator actor named =star-runtime-coordinator=
- one native actor for each selected native Star-Lang actor contract

The coordinator serializes runtime result handling. Native actors execute domain handlers but do not publish transport messages or settle broker deliveries directly.

* Runtime Jobs

A dispatcher proxy creates a runtime job containing:

- deterministic runtime job ID
- target actor name
- original Star-Lang command envelope

It stores the job before calling =tell=. The native actor receives:

#+begin_src lisp
(:kind :star-runtime-job
 :job-id "runtime-job-000001"
 :actor-name "fec-native-importer"
 :command <command-envelope>)
#+end_src

After successful =tell=, the proxy returns =defer-dispatch=. The deterministic dispatcher publishes its accepted lifecycle acknowledgement, records the command as in progress, and the transport adapter keeps the source delivery unsettled.

* Runtime Results

A native handler returns one existing dispatcher result:

- =complete-dispatch=
- =retry-dispatch=
- =fail-dispatch=

The actor wraps that result and asynchronously tells the coordinator:

#+begin_src lisp
(:kind :star-runtime-result
 :job-id "runtime-job-000001"
 :result <dispatch-result>)
#+end_src

The coordinator resolves the original command and invokes =finish-deferred-dispatch=. That operation may:

- produce a typed reply and completed acknowledgement
- produce a retry acknowledgement and next attempt
- produce a retryable or terminal error
- report =:late-terminal= when cancellation already completed the command

The coordinator hands emitted lifecycle outcomes to the existing publication outbox. The source delivery is settled only after those outcomes publish.

* Tell Failure Before Actor Acceptance

A failure while telling the native actor occurs before the actor mailbox accepted the job.

The proxy removes the runtime job and returns an explicit =retry-dispatch= result with a configured delay. The normal dispatcher and transport layers publish accepted and retry acknowledgements and schedule a new Star-Lang attempt.

This is a runtime availability failure, not a malformed Star-Lang command.

* Handler Failure

A Common Lisp condition escaping the registered native handler becomes:

#+begin_src lisp
(fail-dispatch
 :code "star.native-handler-error"
 :message <condition-text>
 :retryable nil)
#+end_src

This prototype uses a terminal default because it cannot infer whether an arbitrary domain condition is safe to retry. Later actor declarations may add typed condition-to-restart policy.

* Cancellation Race

Cancellation is cooperative. A command may become terminal while its actor message is already queued or executing.

The cancellation path:

1. publishes the terminal cancellation error
2. settles the held source delivery and cancellation control delivery
3. leaves the runtime job until the actor result arrives
4. classifies the later result as =:late-terminal=
5. clears the runtime job without publishing new outcomes or reopening the command

The terminal dispatcher record remains authoritative.

* Shutdown

Optional adapter shutdown:

1. stops every actor created by the adapter
2. stops the coordinator actor
3. invokes runtime-system shutdown

Stopping the cl-gserver adapter does not invalidate the Star-Lang manifest. The same contracts may later be executed in standalone mode or through another adapter.

* Conformance Tests

The execution-mode and injected-mailbox suites check:

1. a native Star-Lang actor completes directly through an ordinary handler with no cl-gserver facade
2. standalone execution emits the same accepted, reply, and completed lifecycle sequence
3. runtime liveness snapshots are optional and validated separately from execution
4. coordinator and native actors are constructed through =actor-of= when the adapter is selected
5. adapter command handling performs asynchronous =tell=
6. accepted publishes while the source delivery remains held
7. native completion publishes reply and completed before source acknowledgement
8. native retry creates a delayed next Star-Lang attempt
9. a failed initial =tell= becomes explicit retry without a queued actor message
10. native handler conditions become stable terminal errors
11. cancellation settles transport work and ignores the later actor result
12. runtime jobs clear after completion or late-terminal handling
13. shutdown stops adapter actors and the runtime context
14. native contracts without registered handlers are rejected

* Boundary

This pass does not:

- make cl-gserver mandatory for Star-Lang execution
- load Sento or cl-gserver in continuous integration
- create operating-system threads
- implement actor supervision trees
- map Star-Lang restart declarations to Sento supervision policy
- enforce mailbox bounds or backpressure
- persist runtime jobs, dispatcher records, or publication outboxes
- implement distributed actor discovery
- run external-language actors
- promote Common Lisp as the final selected Star-Lang implementation

The next pass should keep standalone execution as the baseline while adding supervision, mailbox pressure, durable runtime-job recovery, and a concrete Sento smoke adapter as optional runtime capabilities.
