* Product Boundary

Quasar is a basic typed graph editor at its core.

Its first responsibility is to let a user create, inspect, edit, connect, arrange, save, load, import, and export typed nodes and edges. Star-Lang adds transforms and automation to this graph editor. It does not require Quasar to become a distributed actor platform, a domain-server coordinator, or a second StarIntel runtime.

The MVP must remain useful when every transform runs locally in one process and no domain server exists.

The core product is:

#+begin_quote
A local-first graph document editor with typed nodes, typed edges, extensible actions, and a direct path from Star-Lang transform output into the visible graph.
#+end_quote

* Non-Goals

The Quasar core does not require:

- one actor per node or edge;
- one domain server per intelligence discipline;
- distributed supervision;
- remote worker discovery;
- durable distributed transactions;
- an ontology service separate from the graph type registry;
- a workflow engine before basic editing works;
- a canonical intelligence database inside the client;
- a full case-management suite;
- Palantir-scale object permission propagation;
- mandatory evidence, claim, or provenance objects for every manually created graph element.

Those capabilities may integrate later where they solve a measured problem. They are not prerequisites for the editor.

* Core Mental Model

A Quasar workspace contains one or more graph documents.

A graph document contains:

- nodes;
- edges;
- graph metadata;
- layout state;
- type-library references;
- optional action-run records;
- optional source and provenance references;
- view state required to reopen the graph.

The graph document is the product-level unit of editing, undo, save, load, export, and collaboration.

* Minimal Graph Model

** Node

A node contains:

- stable =id=;
- =type=;
- human-readable =label=;
- property map;
- optional source reference;
- optional provenance metadata;
- visual position;
- optional style override;
- creation and update timestamps.

Example:

#+begin_src json
{
  "id": "node-01JZ...",
  "type": "star.core/domain@1",
  "label": "example.org",
  "properties": {
    "name": "example.org"
  },
  "position": {
    "x": 420,
    "y": 240
  }
}
#+end_src

** Edge

An edge contains:

- stable =id=;
- =type= or predicate;
- source node identifier;
- destination node identifier;
- property map;
- optional source reference;
- optional provenance metadata;
- optional style override;
- creation and update timestamps.

Example:

#+begin_src json
{
  "id": "edge-01JZ...",
  "type": "star.core/resolves-to@1",
  "source": "node-domain",
  "destination": "node-ip",
  "properties": {}
}
#+end_src

** Graph metadata

Graph metadata contains:

- graph identifier;
- title;
- description;
- format version;
- created and updated times;
- imported Star-Lang type-library digests;
- layout name;
- viewport state;
- selected style sheet;
- optional workspace tags.

* Primitive Editor Operations

The editor core exposes a deliberately small command set:

1. =add-node=
2. =update-node=
3. =remove-node=
4. =add-edge=
5. =update-edge=
6. =remove-edge=
7. =move-nodes=
8. =set-view=
9. =apply-batch=
10. =undo= and =redo=

Everything else composes these operations.

A transform that discovers five hosts does not mutate Cytoscape directly. It returns a batch containing node and edge additions. The graph editor validates and applies that batch through the same command path used by manual editing and import.

* Architecture

The minimum architecture has four parts:

#+begin_src text
Browser client
  React workbench
  Cytoscape graph adapter
  inspectors and command surface
          |
          | typed HTTP / WebSocket messages
          v
Quasar service
  graph-document CRUD
  validation
  undoable command application
  action registry
  action invocation
          |
          +--> local Star-Lang transform runner
          +--> optional external process adapter
          +--> optional existing StarIntel service
          |
          v
Graph-document storage
#+end_src

No domain server is required in this path.

The Quasar service may initially live inside the existing Common Lisp HTTP process. It should remain a normal library boundary, not a new platform layer.

* Client Responsibilities

The browser client owns:

- graph rendering;
- selection;
- drag, connect, delete, and edit gestures;
- viewport and panel state;
- local preview of pending edits;
- command palette and context menus;
- layout invocation;
- property inspectors;
- undo and redo controls;
- import and export interaction;
- display of action progress and results.

The client does not own the durable graph document when a backend is present.

* Quasar Service Responsibilities

The Quasar service owns:

- graph-document creation, loading, saving, and deletion;
- graph command validation;
- type validation;
- sequential graph revision numbers;
- optimistic-concurrency checks;
- action discovery;
- transform invocation;
- transform-output validation;
- application of transform batches;
- action cancellation where the runner supports it;
- graph import and export;
- live graph-change events.

This is a thin application service. It is not a domain-gserver supervisor.

* Persistence

The storage interface is deliberately narrow:

- =create-graph=
- =load-graph=
- =save-graph=
- =delete-graph=
- =list-graphs=
- =append-command=
- =load-command-history=

MVP storage may be implemented with a single local transactional database or a compact graph-document file plus command log. The editor must not depend on distributed storage.

A practical first implementation may store:

- current graph snapshot;
- monotonically increasing revision;
- bounded undo history;
- optional action-run metadata.

Full event sourcing is not required. An append-only command log is useful only where it directly enables undo, crash recovery, or debugging.

* Revision and Conflict Model

Every accepted graph command advances one graph revision.

A client sends:

- graph identifier;
- expected revision;
- command identifier;
- command payload.

The service either:

- applies the command and returns the new revision;
- rejects malformed or type-invalid content;
- rejects a stale revision and returns the current revision.

For the single-user MVP, this is enough. Collaborative conflict-free replicated data types are deferred until simultaneous editing is a real requirement.

* Star-Lang Integration

Star-Lang defines optional graph actions.

A graph action declares:

- stable action identifier;
- version;
- accepted selection shape;
- parameter schema;
- output graph-operation schema;
- execution adapter;
- optional capability requirements;
- optional cancellation support.

Example declaration:

#+begin_src lisp
(define-graph-action expand-domain
  (:version 1)
  (:accepts ((node star.core/domain@1)))
  (:parameters ())
  (:returns graph-batch)
  (:run dns-expand))
#+end_src

The essential output is a graph batch:

#+begin_src lisp
(graph-batch
  (add-node :id "ip-203.0.113.10"
            :type 'star.core/ip-address@1
            :label "203.0.113.10")
  (add-edge :type 'star.core/resolves-to@1
            :source selected-domain
            :destination "ip-203.0.113.10"))
#+end_src

The graph editor validates and applies this batch. Star-Lang does not receive direct access to the Cytoscape instance.

* Action Execution Adapters

The MVP needs one execution interface and may provide several adapters:

1. in-process Common Lisp function;
2. supervised local subprocess;
3. HTTP service;
4. existing StarIntel actor or domain service.

The first adapter is sufficient to ship the graph editor.

Domain servers remain an optional adapter for transforms that already live behind one. Quasar does not create domain servers merely to invoke an action.

* Type Registry

The editor needs a small type registry, not a universal ontology platform.

A type definition supplies:

- canonical type identifier;
- node or edge kind;
- display name;
- icon or shape hint;
- property definitions;
- label template;
- allowed edge endpoints when constrained;
- optional default actions.

Star-Lang libraries may contribute these type definitions. The active graph records the library identifiers and digests it uses.

Unknown types remain loadable as generic nodes or edges so a missing extension does not destroy a graph.

* Action Discovery

Context-menu and command-palette actions are computed from:

- selected node and edge types;
- action acceptance declarations;
- currently available adapters;
- user-provided parameters;
- optional capabilities.

The action registry returns a flat ranked list. It does not require a domain-server routing tree.

* Undo and Redo

Manual edits and transform result batches use the same reversible graph-command representation.

A command records enough information to derive its inverse:

- adding a node is undone by removing it and any edges created in the same batch;
- removing a node stores the removed node and attached edges;
- updating properties stores previous values;
- moving nodes stores previous positions;
- applying a transform batch is undone as one group.

External side effects are not undone by graph undo. Only graph mutations are reverted. A transform that scans a host has already performed that scan; undo removes its graph output.

* Import and Export

MVP formats:

- canonical Quasar JSON graph document;
- canonical Star-Lang graph batch;
- GraphML import and export where type information can be preserved;
- CSV node and edge import through explicit mappings.

Exports preserve stable identifiers, types, properties, positions, and library references.

* Live Updates

WebSocket support is limited to:

- graph command accepted;
- graph batch applied;
- action started;
- action progress;
- action completed;
- action failed;
- action cancelled.

Reconnect uses the graph revision:

1. client reconnects with its last revision;
2. service sends missed accepted commands when retained;
3. otherwise service sends the current graph snapshot.

This does not require a general actor-event bus.

* Security Boundary

Star-Lang actions and extensions do not run inside the browser.

Untrusted local actions should run in a subprocess with explicit capabilities. The graph editor validates all returned graph operations before applying them.

Capabilities may limit:

- filesystem paths;
- network access;
- executable invocation;
- target scope;
- graph mutation count;
- runtime;
- output size.

The capability model belongs to the action runner. It does not force the graph editor into a distributed service architecture.

* MVP Slice

The first executable slice is:

1. open a blank graph;
2. add two typed nodes;
3. connect them with a typed edge;
4. edit labels and properties;
5. drag and lay out nodes;
6. save and reopen the graph;
7. undo and redo edits;
8. load one Star-Lang type library;
9. invoke one in-process Star-Lang graph action;
10. stream and apply its returned graph batch;
11. export the graph as canonical JSON.

If this slice requires domain servers, the design has failed its simplicity requirement.

* Rejected Core Complexity

| Proposal | Core decision |
|----------+---------------|
| Domain server per transform family | rejected; optional integration only |
| Actor per graph object | rejected |
| Distributed workflow runtime before editor | rejected |
| Mandatory claim/evidence object for every edit | rejected |
| Separate graph database requirement | rejected for MVP |
| GraphQL schema | rejected |
| Node.js backend | rejected |
| Client-side canonical database | rejected |
| CRDT collaboration in first release | deferred |
| Full event sourcing | deferred unless recovery demands it |

* Implementation Order

** Phase 0: graph document

- graph model;
- command model;
- in-memory store;
- JSON import and export;
- unit tests for add, update, remove, batch, undo, and revision checks.

** Phase 1: editor

- React shell;
- Cytoscape adapter;
- add, edit, connect, move, delete;
- inspector;
- save and reopen;
- undo and redo.

** Phase 2: typed actions

- Star-Lang type-library loading;
- action registry;
- one in-process action adapter;
- graph-batch validation;
- progress and cancellation UI.

** Phase 3: useful integrations

- subprocess action adapter;
- StarIntel search and import adapter;
- evidence and provenance metadata where supplied;
- optional domain-service adapter for existing services.

** Phase 4: measured scale

- paged projections only when graph sizes demand them;
- collaboration only when simultaneous editing is required;
- alternate large-graph renderer only after Cytoscape limits are measured.

* Acceptance Tests

The architecture is accepted when:

- a graph can be edited without StarIntel running;
- a graph can be edited without any domain server;
- all manual graph edits pass through one command interface;
- Star-Lang actions return graph batches rather than renderer mutations;
- transform output can be undone as one graph mutation group;
- unknown extension types remain visible and round-trip safely;
- reconnect restores the current graph by revision or snapshot;
- the first working vertical slice fits inside the client, Quasar core, and one local action runner.
