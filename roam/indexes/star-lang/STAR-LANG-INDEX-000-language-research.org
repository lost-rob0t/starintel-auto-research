* Purpose

This project index[fn:project-index] tracks Star-Lang[fn:star-lang] research and design while keeping the Common Lisp[fn:common-lisp] implementation path separate from approval and the repository implementation slot.

* Canonical Source

- [[file:../../design/STAR-00-Starintel-4th-generation-Ideas.org][STAR-00 StarIntel Fourth Generation Ideas]]

The source backlog remains the canonical inventory. The current instruction is to continue the Common Lisp language until it reaches the design acceptance criteria before returning to Racket[fn:racket] and Scheme[fn:scheme] comparison work.

* Design Candidate

- [[file:../../design/star-lang/STAR-LANG-001-common-lisp-language-runtime.org][STAR-LANG-001 Common Lisp Language and Durable Runtime]]

Status: draft and pending nsaspy approval. The design freezes the compiler, source grammar, checked intermediate representation[fn:ir], versioned schemas, plan hashing, deterministic runtime, event history, replay, actor boundary, recovery model, package layout, testing layers, and implementation phases.

The design does not enter =roam/implement/= until it is explicitly approved and the one-design implementation slot is available.

The foundation research defines the bounded language and durable execution requirements. The baseline proves the required email-enumeration workflow can be expressed and tested in Common Lisp without reader macros or unrestricted =EVAL=.

* Active Common Lisp Core

The next implementation-oriented research slice adds:

- a local ASDF[fn:asdf] system with no Quicklisp requirement
- a bounded Star-Lang tokenizer and parser
- source positions and spans
- versioned schemas and content-addressed documents
- SHA-256 plan and document identity
- checked effect declarations
- stable plan-node identifiers
- branches and checkpoints
- append-only in-memory run events
- declared agent-tool enforcement
- live execution and replay without redispatching recorded actor or agent results
- negative parser, compiler, schema, persistence, branch, hashing, and replay tests

* Ordered Loops

** DONE Loop 000: Language Design Foundations

Define the minimum safe source, checked intermediate representation, effects, durable execution, recovery, actor, reasoning, and testing architecture.

** DONE Loop 001: Common Lisp Baseline

Implement the required email-enumeration workflow with ordinary macros and no reader macros or unrestricted =EVAL=.

** RESEARCHING Loop 002: Common Lisp Compiler and Reference Runtime

Implement the source reader, source spans, versioned schemas, stable identifiers, plan hashes, checked effects, branches, events, and effect-safe replay through a local ASDF system.

** TODO Loop 003: Durable Recovery and Adapters

Implement journal and checkpoint protocols, crash-after-every-event testing, conditions and durable restarts, human decisions, Sento adapters, Star Router or ZeroMQ adapters where required, and Prolog capability routing.

** TODO Loop 004: Tooling and Measured Optimization

Implement command-line and Emacs interfaces, source-mapped diagnostics, trace and explanation views, phase caches, bounded parallel scheduling, and differential tests.

** TODO Loop 005: Comparative Evaluation

After the Common Lisp path satisfies its acceptance criteria, implement equivalent Racket and Scheme fixtures and record the full cross-language comparison without discarding the completed Common Lisp design.

* Common Lisp Readiness Gate

The Common Lisp path is not ready merely because the email fixture passes. It must satisfy the acceptance criteria in =STAR-LANG-001=, including:

1. safe source parsing without the Common Lisp reader
2. source maps through compilation
3. versioned schemas and capabilities
4. reproducible SHA-256 plan hashes
5. checked effects and capability authority
6. deterministic event history
7. replay without duplicate effects
8. checkpoint-independent recoverability
9. typed conditions and durable restarts
10. actor and reasoning adapter contracts
11. crash-boundary tests
12. public tooling and diagnostics
13. local ASDF loading and testing without Quicklisp

* Footnotes and Glossary

[fn:project-index] Project index: A maintained navigation file that lists authoritative notes, current work, dependencies, ordering, and decision gates for one project area.

[fn:star-lang] Star-Lang: The proposed StarIntel language for declaring typed document flows, actor work, reasoning, persistence, recovery, and resumable analysis behavior.

[fn:common-lisp] Common Lisp: A standardized Lisp programming language with functions, macros, objects, generic functions, and a structured condition and restart system.

[fn:racket] Racket: A Lisp-family language and language-building platform that supports embedded languages and dedicated =#lang= module languages.

[fn:scheme] Scheme: A Lisp-family language standardized around a small core, lexical scope, procedures, and macros, with multiple practical implementations.

[fn:ir] Intermediate representation, or IR: A compiler-owned data structure between source syntax and runtime execution.

[fn:asdf] Another System Definition Facility, or ASDF: The standard Common Lisp system loader used to describe source-file order and test operations.

[fn:conformance-fixture] Conformance fixture: One reviewed input and expected-output dataset used unchanged across implementations so their behavior can be compared directly.
